1. Executive Summary
JobPilot is a multi-agent AI system that automates the entire job search lifecycle — from resume upload to interview preparation. It is designed for job seekers who waste hours weekly on manual job searching, resume tailoring, and application tracking with little structure or feedback.

Mission Statement
To give every job seeker an unfair advantage by automating the repetitive parts of job hunting so they can focus on what matters: preparing for interviews and making human connections.

3. Authentication & Onboarding
3.1 Google OAuth Sign-In
Functional Requirements
	•	User signs in exclusively via Google OAuth (no email/password auth in v1)
	•	NextAuth.js v5 handles the OAuth flow
	•	Required OAuth scopes must be requested on first sign-in:
	•	openid, profile, email — basic identity
	•	https://www.googleapis.com/auth/gmail.readonly — read inbox for replies/invites
	•	https://www.googleapis.com/auth/gmail.send — send reminders and prep guides
	•	https://www.googleapis.com/auth/calendar.events — create interview/reminder events
	•	https://www.googleapis.com/auth/contacts.readonly — check existing contacts for referrals
	•	Access token and refresh token stored encrypted in NextAuth JWT session
	•	Token refresh handled automatically by NextAuth before expiry
	•	If user revokes Google access: detect on next API call, prompt re-auth with clear message
	•	First-time users redirected to /onboarding after sign-in
	•	Returning users redirected to /dashboard

UI Requirements
	•	Sign-in page: centered card layout, JobPilot logo at top, single 'Continue with Google' button (44px height min), brief value proposition below button
	•	Google button: use official Google branding (white button, Google logo, 'Continue with Google' text) — do NOT use custom styling that violates Google brand guidelines
	•	Loading state: spinner inside button, button disabled, text changes to 'Signing in...'
	•	Error state: red inline error message below button — e.g. 'Sign-in failed. Please try again.'
	•	Page must have descriptive <title>: 'Sign In — JobPilot'
	•	No other navigation links on sign-in page except Privacy Policy and Terms of Service in footer
	•	Mobile: same card but full-width on screens < 480px

Accessibility Requirements (WCAG 2.1 AA)
	•	Google button must have aria-label='Sign in with Google'
	•	Loading state: aria-busy='true' on button, aria-label changes to 'Signing in, please wait'
	•	Error message: role='alert' and aria-live='polite' so screen readers announce it
	•	Page must have proper <h1> heading: 'Welcome to JobPilot'
	•	Focus must be on the sign-in button on page load
	•	Color contrast: all text must meet WCAG 4.5:1 minimum ratio

Testing Criteria
	•	Click 'Continue with Google' — Google consent screen appears with all required scopes listed
	•	Complete Google auth — user is created in database with correct email/name
	•	First-time user is redirected to /onboarding
	•	Returning user is redirected to /dashboard
	•	Revoke Google access in Google Account settings — next page load shows 're-authenticate' prompt
	•	Token expires — refresh happens silently without user seeing a sign-in prompt
	•	Sign-in page loads in < 2 seconds on 3G connection
	•	Screen reader announces the sign-in button and error states correctly

3.2 Onboarding Flow (4-Step Wizard)
Step 1 — Resume Upload
	•	User uploads PDF or DOCX resume (max 5MB)
	•	File validated client-side: type (pdf/docx only), size (< 5MB)
	•	Progress indicator shows upload + parse status
	•	File uploaded to Google Cloud Storage, URL stored in DB
	•	Resume Agent triggered automatically after upload
	•	Parsed data shown as preview: skills list, experience count, education
	•	User can re-upload if parse looks incorrect
	•	Step cannot be skipped — resume is required

UI Requirements
	•	Step 1 of 4 progress indicator at top (pill steps, current = filled, others = outlined)
	•	Large drag-and-drop zone (dashed border, 200px min-height): 'Drag your resume here or click to browse'
	•	Accepted formats shown below zone: 'PDF or DOCX, max 5MB'
	•	Upload progress bar while uploading
	•	Parse progress: spinner with text 'Reading your resume...'
	•	Success state: green checkmark, filename shown, 'Looks good!' message
	•	Preview card below: shows extracted skills as chips, job count, years of experience
	•	Re-upload button available after successful parse
	•	'Next' button enabled only after successful parse
	•	Error state: red border on zone, error message with specific reason (file too large / wrong format)

Step 2 — Job Preferences
	•	Target roles: multi-tag input, user types role names, autocomplete from common list
	•	Target companies: multi-tag input, type company names; option to toggle 'Open to any company'
	•	Location: text input with autocomplete; include 'Remote' as option
	•	Work type: checkbox group — Remote / Hybrid / On-site (multi-select)
	•	Salary expectation: min salary number input with currency selector (INR/USD default based on location)
	•	Experience level: dropdown — Entry / Mid / Senior / Lead / Principal
	•	All fields optional except at least one target role OR 'Open to any company' toggled

Step 3 — Reminder Preferences
	•	Preferred reminder time: time picker — what time should JobPilot email job reminders?
	•	Reminder timezone: auto-detected from browser, user can change
	•	Reminder frequency: daily / every 2 days / weekly toggle
	•	Preview shown: 'You'll receive job reminders at 8:00 PM IST daily'

Step 4 — Billing Plan Selection
	•	Three cards displayed: Free / Platform Fee (API key required) / Full Service
	•	Free tier: 5 job searches/day, 2 tailored resumes/day, basic tracking — no payment
	•	Platform Fee tier: user provides own Gemini API key, platform charges small monthly fee, unlimited AI calls
	•	Full Service tier: platform provides API key, higher monthly fee, no API key needed
	•	'Start Free' pre-selected by default
	•	Platform Fee and Full Service marked as 'Coming Soon' with join waitlist option
	•	User can skip billing — free tier auto-selected

Testing Criteria
	•	Upload valid PDF resume — parse succeeds and shows extracted data within 10 seconds
	•	Upload file > 5MB — client-side error shown before upload starts
	•	Upload .jpg file — error: 'Only PDF and DOCX files are accepted'
	•	Complete all 4 steps — user record updated with all preferences in DB
	•	Skip Step 2 without selecting a role or toggling 'open to any' — Next button stays disabled
	•	Onboarding can be re-accessed from Settings to update preferences
	•	Step progress persists if user refreshes page mid-onboarding (session storage)
	•	Timezone auto-detected correctly for IST (India), can be changed manually

4. Resume Agent
4.1 Resume Parsing
Functional Requirements
	•	Accept PDF and DOCX resume files
	•	Extract using PyPDF2 for PDF, python-docx for DOCX
	•	Pass extracted text to Vertex AI Gemini with structured extraction prompt
	•	Return structured JSON: skills[], experience[{title,company,duration,bullets[]}], education[{degree,institution,year}], certifications[], languages[]
	•	Handle multi-page resumes (up to 10 pages)
	•	Handle various resume formats: chronological, functional, hybrid, ATS-friendly
	•	If extraction confidence < 70%: flag for manual review, show user what was found
	•	Store raw text AND structured JSON in database
	•	Parsing must complete within 15 seconds

4.2 Career Gap Analysis
	•	Run immediately after resume parsing
	•	Web Search MCP: query '[target role] required skills 2026 job market'
	•	Gemini compares user skills vs market demand
	•	Output: {strong_for: string[], stretch_for: string[], weak_for: string[], missing_skills: [{skill, importance: 'critical'|'important'|'nice_to_have'}]}
	•	Store in resumes.gap_analysis JSONB column
	•	Display on dashboard as insight card — not blocking

4.3 Resume Tailoring
	•	Triggered per job application
	•	Input: user's base resume text + target JD text
	•	Gemini prompt: extract JD keywords, rewrite resume bullets to mirror language, promote matching skills, adjust summary
	•	ATS score calculated: keyword overlap percentage
	•	If ATS score < 70%: iterate up to 3 times
	•	Max 3 iterations regardless — prevent infinite loops
	•	Save tailored PDF to GCS — one per job application
	•	Store GCS URL in applications.tailored_resume_url
	•	Tailoring must complete within 30 seconds

UI Requirements
	•	Resume section on profile page: shows current resume filename, upload date, parsed skills as coloured chips
	•	Career gap card: traffic-light system — green chips = strong skills, amber = gap skills, red chips = missing critical skills
	•	Gap analysis has 'Improve' button next to each missing skill — links to Coursera/Google learning resources
	•	Tailored resume download button on each job card — shows ATS score as percentage badge (green ≥ 70%, amber 50-69%, red < 50%)
	•	Resume preview modal: shows tailored resume text with keywords highlighted in yellow
	•	Loading skeleton while tailoring is in progress — do not block UI
	•	Re-tailor button available if user updates resume or wants a fresh version

Accessibility Requirements (WCAG 2.1 AA)
	•	All resume file inputs must have visible label and aria-label
	•	Skill chips must have role='listitem' and be keyboard-navigable
	•	ATS score badge: aria-label='ATS match score 78 percent'
	•	Loading states must announce to screen readers: aria-live='polite' on status container
	•	File upload drag zone: must work with keyboard (Enter/Space to open file picker)
	•	Career gap chart: provide text alternative for all visual indicators

Performance & Core Web Vitals Requirements
	•	Resume parse + gap analysis: total < 15 seconds (P95)
	•	Resume tailoring per job: < 30 seconds (P95)
	•	PDF file upload: show progress bar for files > 500KB
	•	GCS upload: use signed upload URLs — never proxy large files through backend
	•	Cache gap analysis results for 24 hours — do not re-run on every page load

Testing Criteria
	•	Upload standard chronological PDF resume — all sections correctly extracted
	•	Upload DOCX resume — parsing works correctly
	•	Upload resume with no skills section — graceful fallback to text extraction
	•	ATS tailoring for a React developer job with Python-focused resume — React keywords added, score improves to ≥ 70%
	•	Tailoring completes in < 30 seconds for a 2-page resume
	•	GCS URL is accessible via signed URL (expires after 24h for download, refreshed on request)
	•	Career gap analysis shows different results for 'Frontend Developer' vs 'Data Scientist' roles
	•	Uploading a 4.9MB PDF completes successfully
	•	Uploading a 5.1MB file is rejected client-side with clear error
	•	Re-uploading a new resume replaces the old one and re-runs gap analysis

5. Job Search Agent
5.1 Job Discovery
Functional Requirements
	•	Web Search MCP used to find job listings across: LinkedIn, Indeed, company career pages, Glassdoor
	•	Search queries constructed from user profile: role + location + experience level
	•	If target companies specified: search those first, then broaden
	•	Minimum 10 jobs returned per search; maximum 50
	•	Each job record must contain: title, company, company_domain, jd_summary, apply_url, salary_range (if available), location, posted_date (if available), source_platform
	•	Detect if job is on an ATS platform (Greenhouse, Workday, Lever, iCIMS, Taleo, SmartRecruiters) — store in jobs.is_ats_platform and jobs.ats_platform
	•	Deduplicate: same job from multiple sources = one record (match on company + title)
	•	Cache search results in Upstash Redis — TTL: 2 hours
	•	Search runs maximum once every 30 minutes per user to avoid rate limiting
	•	Store all jobs in DB with status 'found'

5.2 Job Scoring & Ranking
	•	Each job scored 0-100 by Gemini against user's resume + preferences
	•	Scoring factors: skill match (40pts), seniority match (20pts), salary fit (20pts), location preference (10pts), company preference (10pts)
	•	Jobs scored < 40 are hidden by default (accessible via 'Show all' filter)
	•	Jobs sorted by score descending on dashboard
	•	Score stored in jobs.match_score — recalculated if user updates resume

UI Requirements
	•	Job search is triggered from dashboard — 'Find New Jobs' button
	•	Loading state: animated cards (skeleton) showing search in progress, with status text: 'Searching for [Role] jobs...'
	•	Each job card shows: company name, job title, match score badge (colour-coded), location, salary range, 'days ago' posted, source platform logo
	•	Match score badge: green (80-100) = 'Strong Match', amber (60-79) = 'Good Match', gray (40-59) = 'Partial Match'
	•	Filter bar: filter by score range, location, company, ATS platform, date posted
	•	Sort options: Best Match / Newest / Company A-Z
	•	Search status bar: '23 jobs found · Last searched 2 hours ago · Refresh'
	•	Empty state: illustration + 'No jobs found. Try broadening your search criteria.' with Settings link
	•	Pagination: load more button (not infinite scroll — better for accessibility and performance)

Accessibility Requirements (WCAG 2.1 AA)
	•	Job cards must be keyboard-navigable — Tab to each card, Enter to expand/view detail
	•	Match score badge: aria-label='Match score: 85 out of 100, Strong Match'
	•	Filter controls: all labels visible and associated with inputs
	•	Loading state: aria-busy='true' on job list container, aria-live='polite' for status updates
	•	Empty state: role='status' on empty state container
	•	Sort/filter changes should announce results count to screen readers: 'Showing 15 of 23 jobs'

Performance & Core Web Vitals Requirements
	•	Job search (10-20 results): complete within 20 seconds (P95)
	•	Job scoring for 20 jobs: complete within 10 seconds (P95)
	•	Job list initial render: < 1 second using cached DB results
	•	Images (company logos): lazy-loaded, WebP format, max 40x40px
	•	Job cards use virtualisation if > 50 items — do not render all DOM nodes
	•	Redis cache hit: return results in < 200ms

Testing Criteria
	•	Search for 'Frontend Developer' in 'Bangalore' — returns ≥ 10 relevant results
	•	Same search within 30 minutes returns cached results without hitting web search
	•	Job with Greenhouse apply URL has is_ats_platform=true, ats_platform='greenhouse'
	•	Duplicate job from LinkedIn and Indeed — only one record created in DB
	•	Job score for a perfect-match job is ≥ 80
	•	Job score for a mismatched job (e.g. Python role for a React developer) is ≤ 40
	•	Filter by 'Remote only' — only remote jobs shown
	•	Salary filter 'INR 10L - 20L' — only matching salary range jobs shown
	•	Search with no target companies — broad search returns mixed companies
	•	Search with target companies set — those companies appear first in results

6. Triple Attack Package & Outreach Agent
6.1 Direct Apply Link
	•	Find the exact application form URL — not just the careers homepage
	•	Verify URL is accessible (HTTP GET check — status 200 or 301)
	•	If URL check fails: fall back to careers page URL with note 'Direct link unavailable — search careers page'
	•	Store verified apply_url in jobs table

6.2 LinkedIn Referral Contacts
	•	Web Search MCP: query 'site:linkedin.com/in [company name] [department/function]'
	•	Extract 2-3 LinkedIn profile URLs from search results
	•	Store in contacts table: name (if extractable), linkedin_url, title, company
	•	Each contact card shows: name (if found), LinkedIn URL as clickable link, title at company
	•	Include pre-written referral request message template user can copy
	•	Referral message template: 'Hi [Name], I noticed you work at [Company]. I am applying for the [Role] position and would appreciate a referral. I have [X years] experience in [key skill]. Happy to share my resume — would this be possible?'

6.3 LinkedIn Easy Apply
	•	Web Search MCP: check if job is posted on LinkedIn with Easy Apply
	•	If found: store linkedin_url and set easy_apply=true in jobs table
	•	Show as third option on job card: 'Apply via LinkedIn Easy Apply'

6.4 Existing Network Check
	•	Gmail MCP: scan user's contacts for emails matching the company domain
	•	Match logic: extract domain from jobs.company_domain, search contacts for @company_domain emails
	•	If match found: create contact record with is_existing_contact=true
	•	Show prominently on job card: 'You know someone at [Company]! Ask for a referral.' with contact name if available
	•	This check runs once per job when job card is first viewed

UI Requirements
	•	Job detail page has three distinct CTA sections in a card:
	•	Section 1 — Direct Apply: primary blue button 'Apply on [Company] website →', shows company domain
	•	Section 2 — Get a Referral: lists 2-3 LinkedIn contact cards, each with avatar placeholder, name/title, 'View on LinkedIn' button; copy-paste referral message below
	•	Section 3 — LinkedIn Easy Apply: secondary button 'Apply via LinkedIn Easy Apply' (shown only if available)
	•	Network match alert (if existing contact found): amber banner at top of job detail: '⭐ You already know someone at this company — ask them for a referral first!'
	•	All external links open in new tab with rel='noopener noreferrer'
	•	Referral message: textarea (pre-filled, editable), 'Copy Message' button with copy-to-clipboard feedback
	•	Mobile: sections stack vertically, buttons are full width

Accessibility Requirements (WCAG 2.1 AA)
	•	All external links include aria-label describing destination: aria-label='Apply for [Role] on [Company] website, opens in new tab'
	•	Referral message copy button: aria-label='Copy referral message to clipboard', announce success: 'Copied!'
	•	LinkedIn contact cards: role='article' with aria-label='LinkedIn contact at [Company]'
	•	Network match alert: role='alert' so screen readers announce it immediately

Testing Criteria
	•	Job at Google — direct apply URL points to Google's careers application form, not just careers.google.com homepage
	•	Job at Greenhouse-hosted company — apply URL is verified and accessible
	•	Job with inaccessible URL — fallback message shown: 'Search careers page directly'
	•	LinkedIn contacts found for 'Meta, Software Engineer' — 2-3 profile URLs extracted
	•	Company with unusual domain (e.g. mckinsey.com) — correct domain used for network check
	•	User has gmail contact with @google.com email, applies to Google job — network match shown
	•	Referral message template copies to clipboard successfully
	•	Easy Apply job shows third option; non-Easy-Apply job does not show that section

7. Cover Letter Generator
Functional Requirements
	•	Triggered automatically when a job is added to 'To Apply' status
	•	Outreach Agent runs web search for: company overview, recent news (last 3 months), product/service, culture, mission statement
	•	Gemini generates cover letter with:
	•	Opening paragraph: hook referencing specific company news or achievement
	•	Body: 2 paragraphs connecting user's experience directly to JD requirements
	•	Closing: specific enthusiasm for the role and clear call to action
	•	Cover letter must NOT be a generic template — company-specific content required
	•	Word count: 250-350 words
	•	Stored in applications.cover_letter (TEXT column)
	•	User can edit the generated cover letter inline before it is used
	•	Regenerate button available — uses different angle on same research
	•	Download as PDF or copy as plain text

UI Requirements
	•	Cover letter appears on job detail page under 'Your Application' section
	•	Editable textarea showing the generated cover letter — clearly marked as AI-generated
	•	'Edit' button toggles read-only to edit mode
	•	'Regenerate' button with spinner — replaces current content after confirmation dialog
	•	Character/word count shown below textarea
	•	Download PDF and Copy Text buttons side-by-side
	•	Small 'What sources were used?' expandable section showing company research sources
	•	Diff view available: 'What changed?' shows differences between original and edited version

Testing Criteria
	•	Cover letter for Google Cloud job mentions Google Cloud specifically, not just 'Google'
	•	Cover letter references at least one piece of recent company news
	•	Word count is between 250 and 350 words
	•	Regenerated cover letter is meaningfully different from the first version
	•	User edits cover letter — changes are saved on blur (auto-save)
	•	PDF download produces a clean, readable single-page cover letter
	•	Cover letter generation completes within 20 seconds

14. Global Accessibility Requirements (WCAG 2.1 AA)
14.1 Colour & Contrast
	•	All text must meet WCAG 4.5:1 contrast ratio minimum (normal text)
	•	Large text (18px+ or 14px+ bold) must meet 3:1 contrast ratio minimum
	•	Interactive elements (buttons, links) must meet 3:1 contrast against adjacent colours
	•	Do not use colour alone to convey information — always combine with text label or icon
	•	Status badges (match score colours) must include text label — not just colour
	•	Focus indicators: visible 2px outline on all interactive elements — never remove focus outline

14.2 Keyboard Navigation
	•	ALL interactive elements reachable by Tab key
	•	Logical tab order: left-to-right, top-to-bottom
	•	Skip to main content link as first focusable element on every page
	•	Modal dialogs: trap focus inside modal, return focus to trigger on close
	•	Dropdown menus: arrow keys to navigate, Escape to close
	•	Kanban drag-and-drop: keyboard alternative — select card with Enter, move with arrow keys, confirm with Enter
	•	No keyboard traps — user can always Tab away from any component

14.3 Screen Reader Support
	•	All images have descriptive alt text (decorative images: alt='')
	•	Form inputs: always have visible label OR aria-label
	•	Error messages: role='alert', associated with relevant input via aria-describedby
	•	Loading states: aria-live='polite' for non-urgent updates, aria-live='assertive' for errors
	•	Dynamic content changes: aria-live regions for job count updates, status changes
	•	Page titles: unique, descriptive <title> for every page — '[Page Name] — JobPilot'
	•	Landmark regions: <header>, <nav>, <main>, <aside>, <footer> — one <main> per page
	•	Headings: logical hierarchy (h1 → h2 → h3) — no skipping levels

14.4 Mobile & Touch Accessibility
	•	Minimum touch target size: 44x44px for all interactive elements
	•	Text resizable up to 200% without horizontal scrolling or content loss
	•	No content relying on hover state — touch devices have no hover
	•	Portrait and landscape orientation both supported
	•	Pinch-to-zoom not disabled (never use user-scalable=no in meta viewport)

14.5 Forms & Inputs
	•	Autocomplete attributes on all relevant inputs (name, email, etc.)
	•	Input validation errors shown inline below the relevant field
	•	Required fields marked with asterisk + 'Required' in form legend
	•	Error summary at top of form for complex forms (multi-field validation)
	•	Success states communicated to screen readers via aria-live

	•	All DB connections use TLS

	•	PII (email, name, resume content): never logged in application logs
	•	Stripe webhook: verify signature with Stripe-Signature header
	•	Rate limiting on all API routes: 100 req/min per user, 20 req/min per IP for unauthenticated
	•	Input validation: Zod schema validation on all API route inputs
	•	SQL injection prevention: Prisma ORM parameterised queries — never string-concatenated SQL

16.3 Privacy & Compliance
	•	Privacy Policy: clearly state what data is collected, why, how long retained
	•	Terms of Service: user agreement before first use
	•	GDPR: right to access, right to delete, data portability — all implemented (Section 13.2)
	•	India DPDP Act compliance: data processing consent, data minimisation, purpose limitation
	•	Gmail access: minimal scopes requested — only read and send, not delete or modify
	•	Resume content: never used to train AI models — stated in Privacy Policy
	•	Cookies: only essential cookies without explicit consent; analytics cookies require consent banner

19. Complete Database Schema (Prisma)
All tables: UUID primary keys, timestamps in UTC, soft deletes via is_deleted boolean where applicable.

Key Design Decisions
	•	JSONB columns for flexible structured data (parsed resume, questions, gap analysis) — avoids over-normalisation for hackathon speed
	•	ENUM types for status fields — enforced at DB level, not just application layer
	•	Foreign key constraints on all relationships — data integrity enforced by DB
	•	Indexes on user_id (all tables), status (applications, jobs), created_at (jobs, applications)
	•	Separate reminders table — decoupled from application flow, allows flexible scheduling

Appendix: Glossary
Term
Definition
ATS
Applicant Tracking System — software used by companies to manage job applications. Common platforms: Greenhouse, Workday, Lever.
ATS Score
Keyword match percentage between a resume and job description. Higher score = resume more likely to pass automated screening.
BYO API Key
Bring Your Own API Key — Platform Fee tier where user provides their own Gemini API key.
Triple Attack
JobPilot's 3-path apply strategy per job: direct apply link + LinkedIn referral contacts + LinkedIn Easy Apply link.
Coordinator Agent
The root Google ADK agent that receives all user requests and delegates to specialist sub-agents.
MCP
Model Context Protocol — Anthropic's open standard for AI tool integration (Gmail, Calendar, Web Search).
LCP
Largest Contentful Paint — Core Web Vital measuring when the main content loads.
INP
Interaction to Next Paint — Core Web Vital measuring responsiveness to user interactions.
CLS
Cumulative Layout Shift — Core Web Vital measuring visual stability (no unexpected layout jumps).
WCAG 2.1 AA
Web Content Accessibility Guidelines version 2.1, Level AA — the industry standard accessibility compliance level.
P0 / P1 / P2
Priority levels: P0 = must have for MVP, P1 = important but not blocking, P2 = future enhancement.
